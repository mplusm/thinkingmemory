# ThinkingMemory — Complete Integration Reference

**Version:** 3.0
**Base URL (cloud):** `https://memory.thinkingdbx.com`
**Base URL (local):** `http://localhost:8091`
**Auth (cloud):** `X-API-Key: tm_live_xxx` header required on all requests
**Tenancy (open-source):** optional `X-Tenant-ID` header scopes a request to a tenant; omit it for single-tenant mode
**Embeddings:** generated **server-side** (default: local `bge-small-en-v1.5`, 384-dim); callers send text, not vectors
**MCP:** an MCP server (`thinkingmemory-mcp`) exposes the same operations as agent tools — see [MCP Server](#mcp-server)
**Interactive docs:** `{base_url}/docs` (Swagger UI)

---

> ## ⚠️ v3.0 — the unified memory database (`/v1`)
>
> The four separate layers (working/episodic/semantic/procedural) have been
> **collapsed into one `Memory` substrate** queried by a single primitive,
> **`recall`**: intent in, a ranked + token-budget-packed + cited context out.
> The `mtype` field (`episodic|semantic|procedural|working`) is now a tag on a
> row, not a separate table. Embeddings are generated server-side.
>
> **Use the `/v1` API below for new work.** The detailed per-layer sections
> further down describe the **legacy** API, retained only for migration
> reference (`scripts/migrate_to_memory_db.py` ports old rows into `memory`).

## The unified API (`/v1`)

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `POST` | `/v1/remember` | Store a memory; `text` is embedded server-side |
| `POST` | `/v1/remember/batch` | Store many with one batched embed call |
| `POST` | `/v1/recall` | **The primitive** — intent → packed, cited context |
| `POST` | `/v1/forget` | Forget (`hard=false` soft-closes; `true` deletes) |
| `GET`  | `/v1/memory/{id}` | Fetch one memory |
| `GET`  | `/v1/trace/{id}` | Provenance — why a memory is known |

**Remember**
```bash
curl -X POST {base_url}/v1/remember -H 'Content-Type: application/json' -d '{
  "agent_id": "agent-1",
  "content": {"text": "Q3 revenue target is $4.2M, up 18% from Q2."},
  "mtype": "semantic"
}'
```

**Recall** (intent in; ranked, deduped, packed-to-budget context out)
```bash
curl -X POST {base_url}/v1/recall -H 'Content-Type: application/json' -d '{
  "agent_id": "agent-1",
  "intent": "what is our Q3 revenue goal?",
  "token_budget": 2000,
  "mtypes": ["semantic"]
}'
```
```json
{
  "context": "[1] Q3 revenue target is $4.2M, up 18% from Q2.",
  "items": [{"citation": 1, "id": 7, "mtype": "semantic", "score": 0.81, "why": ["vector","keyword"]}],
  "tokens_used": 16,
  "tokens_saved_vs_dump": 290
}
```

Recall fuses vector + keyword + recency (RRF, salience-weighted) and packs the
top results into `token_budget`. Each recall boosts the salience of what it
surfaced. Working memory (Redis TTL scratchpad) remains under `/working`.

## Architecture Overview

ThinkingMemory provides four memory layers, each serving a distinct purpose:

```
AI Agent (any framework)
│
▼
ThinkingMemory API (FastAPI)
│
├── Working Memory ──────── Redis (TTL-based, volatile)
├── Episodic Memory ─────── PostgreSQL + pgvector (experience log)
├── Semantic Memory ─────── PostgreSQL + pgvector (facts & knowledge)
└── Procedural Memory ───── PostgreSQL + pgvector (skills & habits)
```

All layers are scoped by `agent_id`. In multi-tenant (cloud) mode, all data is additionally isolated by `tenant_id`.

### Tech Stack
- **API:** Python 3.11+ / FastAPI
- **Persistent storage:** PostgreSQL 16+ with pgvector extension
- **Volatile storage:** Redis 7+
- **Embeddings:** Any provider (stored as vectors, not generated by the platform). Columns are a **fixed dimension** (`EMBEDDING_DIM`, default 1536) with an **HNSW index** per layer for fast similarity search. Requests whose embedding length differs from `EMBEDDING_DIM` are rejected with `422`.

---

## Layer Summary

| Layer | Storage | Purpose | Key Characteristics |
|-------|---------|---------|---------------------|
| **Working** | Redis | Short-term reasoning context | Key-value with TTL, auto-expires, volatile |
| **Episodic** | PostgreSQL | Past actions, decisions, outcomes | Append-only, access tracking, compression, similarity search |
| **Semantic** | PostgreSQL | Facts, DB schemas, knowledge | Confidence scoring, lineage tracking, structured knowledge |
| **Procedural** | PostgreSQL | Workflows, preferences, habits | Success/failure tracking, upsert semantics, versioned steps |

### Database Tables

```
Working Memory (Redis — no tables)
  Keys: {tenant_id}:{agent_id}:{key} → JSON value with TTL

Episodic Memory
  memoryitem    — content (JSON), embedding, access_count, is_compressed

Semantic Memory
  fact          — fact text, confidence, source
  datasource    — source_name, source_type, connection_alias, metadata_
  datatable     — table_name, schema_name, table_type, data_source_id (→ datasource)
  datacolumn    — column_name, data_type, data_table_id (→ datatable), lineage
  knowledgeentity — entity_type, name, properties, relationships, confidence

Procedural Memory
  procedure     — name, steps (JSON), success_rate, version
  userpreference — category, key, value, confidence, observation_count
  workflowhabit  — habit_name, pattern (JSON), frequency_count, success_count, failure_count
```

All PostgreSQL tables include: `id` (PK), `tenant_id` (indexed, default `"default"`), `agent_id`, `embedding` (pgvector `vector(EMBEDDING_DIM)` with an HNSW index), `timestamp`.

# Layer 1: Working Memory

Short-term key-value store backed by Redis. Data auto-expires via TTL. Use for reasoning context, intermediate results, and session state that doesn't need to persist beyond the current task.

## Data Model

No database table — stored directly in Redis.

- **Key format:** `{tenant_id}:{agent_id}:{key}` (or `{agent_id}:{key}` in single-tenant mode)
- **Value:** JSON object (any structure)
- **TTL:** Configurable per key, default 300 seconds (5 minutes)

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/working/store` | Store key-value with TTL |
| `GET` | `/working/retrieve/{agent_id}/{key}` | Get a specific key |
| `GET` | `/working/retrieve/{agent_id}` | Get all keys for an agent |
| `GET` | `/working/keys/{agent_id}` | List key names only |
| `PATCH` | `/working/update/{agent_id}/{key}` | Update value (preserves TTL unless overridden) |
| `PATCH` | `/working/extend-ttl/{agent_id}/{key}` | Add seconds to existing TTL |
| `DELETE` | `/working/delete/{agent_id}/{key}` | Delete a specific key |
| `DELETE` | `/working/clear/{agent_id}` | Delete all keys for an agent |
| `GET` | `/working/exists/{agent_id}/{key}` | Check if key exists |
| `GET` | `/working/stats/{agent_id}` | Key count and total size |

### Store

```bash
curl -X POST {base_url}/working/store -H "Content-Type: application/json" -d '{
  "agent_id": "agent-123",
  "key": "current_task",
  "value": {"step": 3, "context": "analyzing sales data", "partial_result": [1, 2, 3]},
  "ttl": 3600
}'
```

**Request schema:**

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `agent_id` | string | **yes** | — | Owning agent |
| `key` | string | **yes** | — | Key name |
| `value` | JSON object | **yes** | — | Any JSON value |
| `ttl` | int | no | `300` | Time-to-live in seconds |

### Retrieve

```bash
# Single key — returns {key, value, ttl}
curl {base_url}/working/retrieve/agent-123/current_task

# All keys — returns {key: value, ...}
curl {base_url}/working/retrieve/agent-123
```

### Update (preserves TTL)

```bash
curl -X PATCH {base_url}/working/update/agent-123/current_task \
  -H "Content-Type: application/json" -d '{
    "value": {"step": 4, "context": "generating report"},
    "ttl": null
  }'
```

If `ttl` is `null`, the remaining TTL is preserved. If set, the TTL is replaced.

### Extend TTL

```bash
curl -X PATCH {base_url}/working/extend-ttl/agent-123/current_task \
  -H "Content-Type: application/json" -d '{"additional_seconds": 600}'
```

### Stats

```bash
curl {base_url}/working/stats/agent-123
```

Response:
```json
{"key_count": 3, "total_size_bytes": 1247, "keys": ["current_task", "scratchpad", "cache_query_1"]}
```

# Layer 2: Episodic Memory

Persistent log of past actions, decisions, and outcomes. Stored in PostgreSQL with vector embeddings for similarity search. Supports access tracking (relevance scoring) and compression of similar memories.

## Data Model — `MemoryItem`

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | int | auto | — | Primary key |
| `tenant_id` | string | no | `"default"` | Tenant isolation |
| `agent_id` | string | **yes** | — | Owning agent |
| `memory_type` | string | no | `"episodic"` | Type tag (also `"episodic_compressed"` for compressed clusters) |
| `content` | JSON object | **yes** | — | The memory content (flexible structure) |
| `embedding` | vector | no | null | For similarity search |
| `timestamp` | datetime | no | now | When the memory was created |
| `extra_data` | JSON object | no | null | Additional metadata |
| `access_count` | int | no | `0` | How many times this memory has been retrieved |
| `last_accessed` | datetime | no | null | When this memory was last retrieved |
| `is_compressed` | bool | no | `false` | Whether this memory has been compressed into a cluster |
| `source_memory_ids` | JSON array | no | null | IDs of original memories (for compressed entries) |

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/memory/store` | Store a new memory |
| `GET` | `/memory/retrieve/{agent_id}?limit=10` | Retrieve recent memories (tracks access) |
| `POST` | `/memory/similar/{agent_id}` | Find similar memories by embedding |
| `DELETE` | `/memory/forget/old/{agent_id}?days=30` | Delete memories older than N days |
| `DELETE` | `/memory/forget/low-relevance/{agent_id}?min_access_count=1&days_since_access=7` | Delete rarely-accessed memories |
| `POST` | `/memory/compress/{agent_id}?similarity_threshold=0.3&min_cluster_size=3` | Compress similar memories into clusters |
| `DELETE` | `/memory/compress/cleanup/{agent_id}` | Delete originals after compression |
| `GET` | `/memory/stats/{agent_id}` | Memory statistics |

### Store

```bash
curl -X POST {base_url}/memory/store -H "Content-Type: application/json" -d '{
  "agent_id": "agent-123",
  "content": {
    "goal": "Analyze Q4 revenue trends",
    "actions": ["queried sales_data table", "grouped by region"],
    "outcome": "Found 15% growth in APAC",
    "reasoning": "Used explicit joins per user preference"
  },
  "extra_data": {"duration_ms": 3400, "tables_accessed": ["sales_data", "regions"]},
  "embedding": [0.1, 0.2, "..."]
}'
```

### Retrieve (with access tracking)

```bash
curl {base_url}/memory/retrieve/agent-123?limit=5
```

Every retrieval increments `access_count` and updates `last_accessed`. This enables relevance-based forgetting.

### Similarity Search

```bash
curl -X POST {base_url}/memory/similar/agent-123 -H "Content-Type: application/json" -d '{
  "embedding": [0.1, 0.2, "..."],
  "limit": 5,
  "similarity_threshold": 0.5
}'
```

### Compression

Compression groups similar memories (by embedding distance) into consolidated entries:

```bash
# Step 1: Compress — creates cluster entries, marks originals
curl -X POST "{base_url}/memory/compress/agent-123?similarity_threshold=0.3&min_cluster_size=3"

# Step 2 (optional): Delete originals to free space
curl -X DELETE {base_url}/memory/compress/cleanup/agent-123
```

Compressed memory content:
```json
{
  "type": "compressed",
  "original_count": 5,
  "summary": [{"goal": "..."}, {"goal": "..."}, "..."],
  "time_range": {"start": "2026-01-01T...", "end": "2026-01-15T..."}
}
```

### Forgetting

```bash
# Delete memories older than 30 days
curl -X DELETE "{base_url}/memory/forget/old/agent-123?days=30"

# Delete memories accessed < 1 time AND not accessed in 7 days
curl -X DELETE "{base_url}/memory/forget/low-relevance/agent-123?min_access_count=1&days_since_access=7"
```

### Stats

```bash
curl {base_url}/memory/stats/agent-123
```

Response:
```json
{
  "total_memories": 142,
  "compressed_memories": 12,
  "active_memories": 130,
  "memories_with_embeddings": 138,
  "average_access_count": 3.7
}
```

# Layer 3: Semantic Memory

Long-lived knowledge storage. Contains both simple text facts and structured models for database schemas, column lineage, and generic knowledge entities.

## Model Guide — When to Use What

| Use Case | Model | Example |
|----------|-------|---------|
| Loose text knowledge | `Fact` | "The sales team uses Salesforce for CRM" |
| Database/warehouse connection | `DataSource` | production_postgres, snowflake_warehouse |
| Table or view metadata | `DataTable` | public.users, analytics.daily_revenue |
| Column definitions + lineage | `DataColumn` | users.email (varchar, not null, PII) |
| API endpoints, configs, concepts | `KnowledgeEntity` | /api/v1/users, redis_config, auth_service |

| `PATCH` | `/semantic/knowledge/update/{knowledge_id}` | Update fields |
| `DELETE` | `/semantic/knowledge/delete/{knowledge_id}` | Delete |
| `DELETE` | `/semantic/knowledge/forget/low-confidence/{agent_id}?confidence_threshold=0.5` | Bulk delete low-confidence |

```bash
curl -X POST {base_url}/semantic/knowledge/store -H "Content-Type: application/json" -d '{
  "agent_id": "agent-123",
  "entity_type": "api_endpoint",
  "name": "/api/v1/users",

# Filter by type
curl "{base_url}/semantic/knowledge/retrieve/agent-123?entity_type=api_endpoint"
```

# Layer 4: Procedural Memory

Behavioral knowledge — how to do things, how the user prefers things, and patterns the agent has observed.

## Model Guide — When to Use What

| Use Case | Model | Example |
|----------|-------|---------|
| Named multi-step workflow | `Procedure` | "deploy_to_staging" with 5 steps |
| User style/behavior preference | `UserPreference` | code_style.indent = spaces_4 |
| Recurring behavioral pattern | `WorkflowHabit` | "user runs standup query every morning" |

## 4a. Procedures (`/procedural/`)

Named step-by-step workflows with success rate tracking.

### Data Model — `Procedure`

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | int | auto | — | Primary key |
| `tenant_id` | string | no | `"default"` | Tenant isolation |
| `agent_id` | string | **yes** | — | Owning agent |
| `name` | string | **yes** | — | Procedure name |
| `description` | string | no | null | What the procedure does |
| `steps` | JSON array | **yes** | — | Ordered list of step objects |
| `embedding` | vector | no | null | For similarity search |
| `success_rate` | float | no | `1.0` | 0.0 to 1.0 |
| `timestamp` | datetime | no | now | Creation time |
| `version` | int | no | `1` | Procedure version |

### API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/procedural/store` | Store a procedure |
| `GET` | `/procedural/retrieve/{agent_id}?limit=10` | List procedures |
| `POST` | `/procedural/similar/{agent_id}` | Find similar by embedding |
| `PATCH` | `/procedural/update/{procedure_id}` | Update success rate |
| `DELETE` | `/procedural/forget/low-success/{agent_id}?success_threshold=0.5` | Bulk delete low-success |

```bash
curl -X POST {base_url}/procedural/store -H "Content-Type: application/json" -d '{
  "agent_id": "agent-123",
  "name": "deploy_to_staging",
  "description": "Full staging deployment pipeline",
  "steps": [
    {"step": 1, "action": "run tests"},
    {"step": 2, "action": "build docker image"},
    {"step": 3, "action": "push to registry"},
    {"step": 4, "action": "kubectl apply"}
  ],
  "success_rate": 0.95,
  "version": 2
}'
```

## 4b. User Preferences (`/procedural/preferences/`)

How the user likes things done. Supports **upsert semantics** — storing a preference with the same `(agent_id, category, key)` updates the existing record and increments `observation_count`.

### Data Model — `UserPreference`

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | int | auto | — | Primary key |
| `tenant_id` | string | no | `"default"` | Tenant isolation |
| `agent_id` | string | **yes** | — | Owning agent |
| `category` | string | **yes** | — | e.g. `"code_style"`, `"communication"`, `"sql_style"` |
| `key` | string | **yes** | — | e.g. `"indent_style"`, `"response_length"` |
| `value` | string | **yes** | — | e.g. `"spaces_4"`, `"concise"` |
| `confidence` | float | no | `1.0` | 1.0 = explicitly stated, lower = inferred |
| `source` | string | no | null | `"explicit"` or `"inferred"` |
| `observation_count` | int | no | `1` | How many times observed |
| `last_observed` | datetime | no | null | When last confirmed |
| `embedding` | vector | no | null | For similarity search |
| `timestamp` | datetime | no | now | Creation time |

### API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/procedural/preferences/store` | Create or update (upsert) |
| `GET` | `/procedural/preferences/retrieve/{agent_id}?limit=10` | List all preferences |
| `GET` | `/procedural/preferences/by-key/{agent_id}?category=X&key=Y` | Get specific preference |
| `POST` | `/procedural/preferences/similar/{agent_id}` | Find similar by embedding |
| `DELETE` | `/procedural/preferences/delete/{preference_id}` | Delete |
| `DELETE` | `/procedural/preferences/forget/low-confidence/{agent_id}?confidence_threshold=0.5` | Bulk delete low-confidence |

```bash
# Store (creates new or upserts existing)
curl -X POST {base_url}/procedural/preferences/store -H "Content-Type: application/json" -d '{
  "agent_id": "agent-123",
  "category": "sql_style",
  "key": "join_style",
  "value": "explicit_join",
  "confidence": 1.0,
  "source": "explicit"
}'

# Lookup specific preference
curl "{base_url}/procedural/preferences/by-key/agent-123?category=sql_style&key=join_style"
```

**Upsert behavior:** If `(agent_id, category, key)` already exists, the store call:
- Updates `value` and `confidence` to the new values
- Increments `observation_count` by 1
- Sets `last_observed` to now
- Updates `source` if provided

**Suggested category conventions:**

| Category | Example Keys | Example Values |
|----------|-------------|----------------|
| `code_style` | `indent_style`, `quote_style`, `naming_convention` | `"spaces_4"`, `"single"`, `"snake_case"` |
| `sql_style` | `join_style`, `cte_preference`, `alias_style` | `"explicit_join"`, `"prefer_cte"`, `"short"` |
| `communication` | `response_length`, `explanation_depth`, `tone` | `"concise"`, `"detailed"`, `"casual"` |
| `visualization` | `chart_library`, `color_palette`, `default_chart` | `"plotly"`, `"corporate_blue"`, `"bar"` |
| `workflow` | `auto_commit`, `test_first`, `pr_size` | `"ask_first"`, `"always"`, `"small"` |

## 4c. Workflow Habits (`/procedural/habits/`)

Observed behavioral patterns. Tracks **separate success and failure counts** (not a rate) to preserve statistical significance — 100/95 is more reliable than 2/2.

### Data Model — `WorkflowHabit`

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | int | auto | — | Primary key |
| `tenant_id` | string | no | `"default"` | Tenant isolation |
| `agent_id` | string | **yes** | — | Owning agent |
| `habit_name` | string | **yes** | — | e.g. `"morning_standup_query"` |
| `description` | string | no | null | Human-readable description |
| `pattern` | JSON object | **yes** | — | Trigger, steps, context |
| `frequency_count` | int | no | `1` | Times observed |
| `last_performed` | datetime | no | null | When last performed |
| `success_count` | int | no | `0` | User accepted result |
| `failure_count` | int | no | `0` | User rejected/corrected |
| `embedding` | vector | no | null | For similarity search |
| `timestamp` | datetime | no | now | Creation time |
| `tags` | JSON array | no | null | e.g. `["daily", "standup"]` |

**Pattern structure:**
```json
{
  "trigger": "9am daily",
  "steps": ["query jira for open tickets", "summarize blockers", "format as standup update"],
  "context": "morning standup preparation"
}
```

### API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/procedural/habits/store` | Create a habit |
| `GET` | `/procedural/habits/retrieve/{agent_id}?limit=10` | List (ordered by frequency desc) |
| `POST` | `/procedural/habits/similar/{agent_id}` | Find similar by embedding |
| `POST` | `/procedural/habits/increment/{habit_id}` | Increment frequency + success/failure |
| `PATCH` | `/procedural/habits/update/{habit_id}` | Update fields |
| `DELETE` | `/procedural/habits/delete/{habit_id}` | Delete |
| `DELETE` | `/procedural/habits/forget/unused/{agent_id}?max_frequency=1` | Bulk delete rarely-used |

```bash
# Store
curl -X POST {base_url}/procedural/habits/store -H "Content-Type: application/json" -d '{
  "agent_id": "agent-123",
  "habit_name": "morning_standup_query",
  "description": "User queries Jira every morning for standup prep",
  "pattern": {
    "trigger": "9am daily",
    "steps": ["query jira", "filter updated 24h", "summarize blockers", "format update"],
    "context": "morning standup preparation"
  },
  "tags": ["daily", "standup", "jira"]
}'

# Track outcome
curl -X POST {base_url}/procedural/habits/increment/1 \
  -H "Content-Type: application/json" -d '{"success": true}'
```

Increment bumps `frequency_count` by 1, increments `success_count` or `failure_count`, and sets `last_performed` to now.

# Cross-Cutting Concerns

## Similarity Search

All PostgreSQL-backed models support vector similarity search with the same request schema:

```bash
POST {base_url}/{layer}/{model}/similar/{agent_id}

{
  "embedding": [0.1, 0.2, 0.3, "..."],
  "limit": 10,
  "similarity_threshold": 0.5
}
```

Available similarity endpoints:
```
POST /memory/similar/{agent_id}
POST /semantic/similar/{agent_id}
POST /semantic/sources/similar/{agent_id}
POST /semantic/tables/similar/{agent_id}
POST /semantic/columns/similar/{agent_id}
POST /semantic/knowledge/similar/{agent_id}
POST /procedural/similar/{agent_id}
POST /procedural/preferences/similar/{agent_id}
POST /procedural/habits/similar/{agent_id}
```

Embeddings must be stored at creation time for similarity search to find them. Records without embeddings are excluded.

## Multi-Tenant Isolation

- **Single-tenant (local):** `tenant_id` defaults to `"default"`, no auth required
- **Multi-tenant (cloud):** `tenant_id` injected from API key, all queries filtered automatically
- **Key format (Redis):** `{tenant_id}:{agent_id}:{key}`
- **Column filter (PostgreSQL):** `WHERE tenant_id = ?` on all queries

Cloud-specific:

# Integration Patterns

### Pattern 1: Catalog a database schema

```
1. POST /semantic/sources/store         → get source_id
2. POST /semantic/tables/store          → for each table, set data_source_id
3. POST /semantic/columns/store         → for each column, set data_table_id
4. GET  /semantic/tables/with-columns/1 → full picture in one call
```

### Pattern 2: Learn user preferences over time

```
1. Agent infers preference:
   POST /procedural/preferences/store  {confidence: 0.7, source: "inferred"}

2. User confirms:
   POST /procedural/preferences/store  {confidence: 1.0, source: "explicit"}
   → same key → upsert, observation_count = 2

3. Before generating output:
   GET /procedural/preferences/by-key/{agent_id}?category=...&key=...
```

### Pattern 3: Track workflow habits with feedback

```
1. POST /procedural/habits/store        → initial observation
2. POST /procedural/habits/increment/1  {"success": true}   → user accepted
3. POST /procedural/habits/increment/1  {"success": false}  → user corrected
4. DELETE /procedural/habits/forget/unused/{agent_id}?max_frequency=1  → cleanup
```

### Pattern 4: Full agent memory lifecycle

```
Working Memory (during task):
  POST /working/store  → stash intermediate results
  GET  /working/retrieve/{agent_id}/{key}  → resume context

Episodic Memory (after task):
  POST /memory/store  → log what happened, decisions made, outcome

Semantic Memory (learned knowledge):
  POST /semantic/knowledge/store  → new facts discovered during task

Procedural Memory (behavioral learning):
  POST /procedural/preferences/store  → preference confirmed
  POST /procedural/habits/increment/1  → pattern observed again

Periodic maintenance:
  POST   /memory/compress/{agent_id}   → compress old episodic memories
  DELETE /memory/forget/old/{agent_id}  → prune ancient memories
  DELETE /semantic/knowledge/forget/low-confidence/{agent_id}  → prune uncertain knowledge
  DELETE /procedural/habits/forget/unused/{agent_id}  → prune unused habits
```

# MCP Server

In addition to the REST API, ThinkingMemory ships a [Model Context Protocol](https://modelcontextprotocol.io)
server so MCP-capable agents (Claude Desktop, IDE assistants, custom agents) can
use memory as native tools. It talks **directly to PostgreSQL/Redis** (no HTTP
hop) and reads the same `.env` (`DATABASE_URL`, `REDIS_URL`, `EMBEDDING_DIM`).

```bash
pip install -e ".[mcp]"      # or: pip install mcp
thinkingmemory-mcp           # runs over stdio; or: python -m thinkingmemory.mcp_server
```

Register with an MCP client (e.g. Claude Desktop):

```json
{
  "mcpServers": {
    "thinkingmemory": {
      "command": "thinkingmemory-mcp",
      "env": {
        "DATABASE_URL": "postgresql://memory_user:your_password@localhost:5432/thinkingmemory",
        "REDIS_URL": "redis://localhost:6379/0",
        "EMBEDDING_DIM": "1536"
      }
    }
  }
}
```

### Tools

| Tool | Purpose |
|------|---------|
| `remember` | Store a memory (any `mtype`); text embedded server-side |
| `recall` | **The primitive** — intent in, ranked + packed + cited context out |
| `remember_fact` | Convenience: store a semantic fact |
| `remember_procedure` | Convenience: store a procedural workflow |
| `get_memory` | Fetch a single memory by id |
| `forget` | Forget a memory (soft by default; hard deletes) |
| `set_working_memory` / `get_working_memory` / `list_working_memory` | Redis TTL scratchpad |

Each tool accepts an optional `tenant_id`; a process-wide default can be set with
`THINKINGMEMORY_TENANT_ID`.

# Migration / Deployment

### Apply changes (no data loss)

```bash
cd /root/thinkingmemory && git pull
source venv/bin/activate
# init_db() also ensures the pgvector extension and creates HNSW indexes
python -c "from thinkingmemory.core.database import init_db; init_db()"
```

If upgrading a database created **before** embedding dimensions were pinned, size
the existing vector columns and build their indexes (non-destructive):

```bash
python scripts/migrate_vector_dims.py
```

For cloud:
```bash
cd /root/thinkingmemory-cloud && git pull
systemctl restart thinkingmemory-cloud   # service listens on port 8093
```

### Full reset (destroys all data)

```bash
python scripts/recreate_db.py
```

Drops and recreates all 9 tables.
